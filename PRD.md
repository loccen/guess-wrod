## 2. 产品目标与默认判断

### 2.1 核心目标

做一个微信内可快速开局的猜词小游戏。用户每次输入一个词，系统只返回这个词与隐藏答案的接近百分比，用户据此继续推理直到猜中或放弃。

### 2.2 MVP 想验证的问题

1. AI 评分是否比传统词向量式反馈更符合中文用户直觉。
2. 用户是否愿意连续玩多局随机题。
3. 在不做复杂社交玩法的前提下，核心循环是否已经足够成立。
4. 模型调用成本、缓存命中率和响应时延是否可控。

### 2.3 首版默认判断

1. V0.1 只做单人随机局，不做每日挑战、好友挑战、排行榜和分享战绩图。
2. V0.1 接微信登录，不做游客模式。
3. 词库首版只包含中文常见具象名词，不包含抽象概念。
4. 前端只展示分数，不展示 AI 解释。
5. 评分协议采用“后端固定答案，模型只负责普通猜词评分”的架构。

### 2.4 V0.1 核心指标

| 指标 | 目标 |
| --- | --- |
| 单局完成率 | >= 40% |
| 平均每局猜测次数 | 8-25 次 |
| 次日留存 | >= 15% |
| 评分合理性 | 人工基准集命中预期区间比例 >= 80% |
| 平均响应时延 | 90% 请求 <= 2 秒 |
| 单局 AI 成本 | 按真实模型单价和实测 token 重新测算并持续监控 |

## 3. 目标用户与非目标用户

### 3.1 目标用户

| 用户类型 | 特征 | 需求 |
| --- | --- | --- |
| 轻度小游戏用户 | 微信内碎片时间娱乐 | 快速开局、规则简单、反馈明确 |
| 文字游戏爱好者 | 喜欢联想和推理 | 评分像人在判断，不像在猜算法 |
| AI 兴趣用户 | 对模型表现有兴趣 | 想体验“AI 打分”的新鲜感 |

### 3.2 非目标用户

1. 追求强竞技、复杂段位或多人对战的用户。
2. 需要系统化词汇训练的教育场景。
3. 期待完全确定性、可穷举规则的传统益智用户。

## 4. V0.1 范围

### 4.1 本期必做

1. 微信登录并建立用户会话。
2. 首页展示玩法说明和“开始一局”入口。
3. 随机开局，后端固定答案。
4. 游戏页输入猜词并获得分数反馈。
5. 展示猜测历史、当前最高分、猜测次数。
6. 猜中后进入结果页。
7. 支持用户放弃并查看答案。
8. 单局缓存和全局缓存。
9. 基础埋点、日志和评分反馈采集。

### 4.2 本期不做

1. 每日挑战。
2. 好友挑战。
3. 排行榜。
4. 分享战绩图。
5. 复杂道具或商业化功能。
6. 可视化后台管理页。

### 4.3 词库范围

1. 仅收录中文常见具象名词。
2. 首批建议 300-500 个词。
3. 避免冷门、敏感、强专业背景、强争议词。
4. 不纳入抽象概念、流行梗、长短句答案。

## 5. 页面与用户流程

### 5.1 总流程

```text
进入小程序
  ↓
微信登录
  ↓
首页
  ↓
点击「开始一局」
  ↓
创建随机局并进入游戏页
  ↓
反复提交猜词
  ↓
猜中 / 放弃 / 过期
  ↓
结果页
```

### 5.2 首页

页面目标：承接登录后的开局动作和规则说明。

页面模块：

| 模块 | 说明 | 优先级 |
| --- | --- | --- |
| 标题区 | 产品名 + 一句话说明 | P0 |
| 开始一局 | 创建随机局 | P0 |
| 玩法说明 | 输入一个词，系统只返回接近百分比 | P0 |
| 最近战绩 | 最近完成局数、最近一次成绩 | P1 |

首页不出现每日挑战、好友挑战、排行榜入口。

### 5.3 游戏页

页面目标：完成猜词主循环。

页面模块：

| 模块 | 说明 | 优先级 |
| --- | --- | --- |
| 输入框 | 输入猜测词 | P0 |
| 提交按钮 | 提交本次猜测 | P0 |
| 历史记录 | 默认按时间倒序展示有效猜词 | P0 |
| 当前最高分 | 展示本局最接近答案的猜词 | P0 |
| 猜测次数 | 只统计有效猜词次数 | P0 |
| 放弃按钮 | 主动结束本局并查看答案 | P0 |
| 规则入口 | 简版玩法说明 | P1 |
| 评分反馈入口 | 内测期可见，允许标记“这次评分不太对” | P1 |

### 5.4 结果页

页面目标：清楚结束本局，并引导继续下一局。

页面模块：

| 模块 | 说明 | 优先级 |
| --- | --- | --- |
| 状态结果 | 猜中 / 放弃 / 已过期 | P0 |
| 正确答案 | 展示标准答案 | P0 |
| 猜测次数 | 展示本局有效猜词次数 | P0 |
| 总耗时 | 展示开始到结束耗时 | P1 |
| 最接近但未命中的词 | 便于复盘 | P1 |
| 再来一局 | 直接开始新随机局 | P0 |

结果页不做好友分享图，不展示 AI 原始解释。

## 6. 核心规则

### 6.1 游戏规则

1. 每局创建时由后端从词库中抽取一个答案，并在整局内固定不变。
2. 用户输入一个猜词，系统返回 0-100 的整数分数。
3. 标准答案或词库显式别名命中时，后端直接判定为 100 分并结束游戏，不调用模型。
4. 其他普通猜词进入评分链路，最终由后端返回统一结果。
5. 用户可以主动放弃；放弃后展示答案，并标记本局结束。

### 6.2 有效猜词与无效提交

有效猜词：

1. 输入格式合法。
2. 未命中敏感词。
3. 当前游戏状态仍为 `playing`。
4. 不是本局重复猜词。
5. 评分链路成功返回可接受结果。

无效提交：

1. 空输入、超长输入、非法字符格式。
2. 敏感词。
3. 已结束游戏继续提交。
4. 重复猜词。
5. 限流命中。
6. 模型超时或系统错误。

默认规则：无效提交不增加 `guess_count`。

### 6.3 猜测上限与过期

1. 单局最多 100 次有效猜词。
2. 第 100 次有效猜词处理完成后仍未猜中时，本次猜词仍计入次数，随后本局结束并标记为 `expired`。
3. 从创建时间起 24 小时内未完成的随机局，也记为 `expired`。

### 6.4 历史记录展示

1. 默认按提交时间倒序展示。
2. 只展示有效猜词。
3. 同一个词在本局内重复提交时，前端仍展示已有结果，但不新增历史项。

## 7. 状态机

### 7.1 状态定义

| 状态 | 含义 |
| --- | --- |
| `playing` | 游戏进行中，可继续提交猜词 |
| `success` | 已猜中，游戏结束 |
| `give_up` | 用户主动放弃，游戏结束 |
| `expired` | 达到猜测上限或超出存活时间，游戏结束 |

### 7.2 状态转换

| 当前状态 | 触发事件 | 下一个状态 | 是否增加 `guess_count` |
| --- | --- | --- | --- |
| `playing` | 创建成功 | `playing` | 否 |
| `playing` | 提交标准答案或显式别名 | `success` | 是 |
| `playing` | 提交普通有效猜词 | `playing` | 是 |
| `playing` | 提交重复猜词 | `playing` | 否 |
| `playing` | 提交敏感词或格式非法 | `playing` | 否 |
| `playing` | AI 超时或系统错误 | `playing` | 否 |
| `playing` | 用户点击放弃 | `give_up` | 否 |
| `playing` | 第 100 次有效猜词处理完成后仍未猜中 | `expired` | 是 |
| `playing` | 超过 24 小时未完成 | `expired` | 否 |
| `success/give_up/expired` | 再次提交猜词 | 原状态不变 | 否 |

## 8. 接口范围

V0.1 对外接口只覆盖以下能力，字段细节以 [scoring-spec.md](scoring-spec.md) 为准：

1. `POST /api/auth/wechat-login`
   - 用微信 `code` 换取用户会话。
2. `POST /api/games`
   - 只允许 `mode=random`。
3. `GET /api/games/{game_id}`
   - 获取当前局状态和有效猜词历史。
4. `POST /api/games/{game_id}/guesses`
   - 提交猜词，返回分数或拒绝原因。
5. `POST /api/games/{game_id}/give-up`
   - 放弃并结束本局。
6. `POST /api/games/{game_id}/feedback`
   - 内测期评分反馈采集。

## 9. 埋点与运营最小集

### 9.1 必须埋点

| 事件名 | 触发时机 | 核心字段 |
| --- | --- | --- |
| `login_success` | 微信登录成功 | `user_id` |
| `game_created` | 成功创建随机局 | `game_id`, `word_id` |
| `guess_submitted` | 提交有效猜词 | `game_id`, `guess_id`, `source`, `score`, `relation_type` |
| `guess_reused` | 命中单局重复猜词缓存 | `game_id`, `normalized_guess` |
| `game_give_up` | 用户放弃 | `game_id`, `guess_count` |
| `game_success` | 猜中结束 | `game_id`, `guess_count`, `duration_ms` |
| `game_expired` | 过期结束 | `game_id`, `reason` |
| `ai_error` | 模型异常或校验失败 | `game_id`, `error_code` |
| `score_feedback_submitted` | 用户反馈评分不合理 | `game_id`, `guess_id` |

### 9.2 V0.1 运营方式

1. 不做后台管理页面。
2. 通过数据库查询、日志和简单脚本查看高频问题。
3. 评分反馈优先服务于内测调优，不追求完整工单系统。

## 10. 风控要求

1. 用户输入不得作为模型指令执行，只能作为普通字符串字段传入。
2. 用户猜词和词库都要经过敏感词过滤。
3. 答案只保存在后端；前端在游戏结束前不能拿到答案。
4. 单用户、单局、单 IP 均需限流。
5. 模型异常、JSON 非法、系统错误时，不消耗本次猜测次数。

## 11. MVP 验收标准

### 11.1 功能验收

| 功能 | 验收标准 |
| --- | --- |
| 微信登录 | 用户能成功换取会话并进入首页 |
| 创建游戏 | 点击“开始一局”后返回 `game_id` 并进入 `playing` |
| 固定答案 | 同一局内答案不变化 |
| 提交猜词 | 有效猜词返回 0-100 分数、`relation_type`、`source` |
| 猜中判断 | 标准答案或显式别名命中时直接返回 100 并结束 |
| 重复猜词 | 返回已有结果，不增加 `guess_count` |
| 放弃 | 用户确认后结束游戏并展示答案 |
| 结果页 | 展示状态、答案、猜测次数、再次开始入口 |

### 11.2 体验验收

| 项目 | 标准 |
| --- | --- |
| 响应速度 | 90% 请求 <= 2 秒 |
| 评分合理性 | 基准集符合预期区间比例 >= 80% |
| 误导控制 | 强搭配词不应普遍高于其关系类型上限 |
| 连续游玩意愿 | 内测用户平均愿意玩 2 局以上 |

### 11.3 技术验收

| 项目 | 标准 |
| --- | --- |
| 答案安全 | 游戏结束前前端无法拿到答案 |
| 日志完整 | 有效猜词、异常和反馈均有记录 |
| 成本可见 | 可按模型、局数、token 统计平均成本 |
| 缓存生效 | 可统计单局缓存与全局缓存命中率 |
| 风控生效 | 限流、敏感词、异常不计次规则都能验证 |

## 12. V0.2 待办

仅当 V0.1 的核心循环、评分手感和成本数据成立后，再进入以下扩展：

1. 每日挑战。
2. 好友挑战。
3. 分享战绩图。
4. 简单排行榜。
5. 主题词库与难度分层。

## 13. 实施顺序建议

1. 先完成微信登录、随机局创建、游戏状态存储。
2. 补齐词库、归一化、精确命中和缓存链路。
3. 接模型评分、后处理和统一错误语义。
4. 做小程序首页、游戏页、结果页。
5. 做埋点、评分反馈和基准集验收。
6. 小范围内测后，再决定是否进入 V0.2 社交玩法。
